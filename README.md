# Beleaf Typography Render Service

Small HTTP service that draws Thai headline + overlay bubbles onto a
product photo using a real headless browser (Playwright/Chromium) and
real Google Fonts, driven by the `Design Template JSON` already stored
in `16_DESIGN_BIBLE`. This replaces the `OpenAI Image Typography Edit`
step in Workflow 3 v13.1 — same inputs (source image + headline +
overlayText + design), but Thai glyphs are always pixel-correct
because they're rendered by the browser's real font engine, not
generated by an AI model guessing shapes.

## Why this exists

`gpt-image-1` image edits look great visually but sometimes distort
Thai vowels/tone marks because it's generating pixels, not placing
real font glyphs. A browser screenshot renderer can't get Thai text
wrong — it's the same rendering path as any Thai website.

## Deploy (Render.com — free/hobby tier is enough for testing)

1. Push this folder to a new GitHub repo (or use Render's "public git
   repo" deploy option pointing at a subfolder).
2. In Render: New -> Web Service -> connect the repo -> Environment:
   **Docker** (it will pick up the included `Dockerfile` automatically).
3. Add an environment variable `RENDER_AUTH_TOKEN` = (any random
   secret string you choose) — this is the bearer token n8n will send.
4. Deploy. Render gives you a public HTTPS URL, e.g.
   `https://beleaf-render.onrender.com`.
5. Test: `curl https://beleaf-render.onrender.com/health` should
   return `{"ok":true}`.

Railway / Fly.io work the same way (Dockerfile-based deploy, add the
same env var).

## API

`POST /render`
Header: `Authorization: Bearer <RENDER_AUTH_TOKEN>`
Body (JSON):
```json
{
  "imageBase64": "<base64 of the source jpg, no data: prefix>",
  "imageMimeType": "image/jpeg",
  "design": { "Font Headline": "...", "Font Body": "...", "Headline Weight": "800",
              "designTemplate": { ...the parsed "Design Template JSON" column... } },
  "headline": "ข้อความพาดหัว",
  "overlayText": ["บับเบิล 1", "บับเบิล 2"],
  "imageSequence": 1,
  "imageCount": 2
}
```
Response: raw `image/png` binary (1080x1080).

## Wiring into Workflow 3

Replace the two nodes `OpenAI Image Typography Edit` +
`Decode AI Edited Image` with a single HTTP Request node:

- Method: POST
- URL: `https://<your-render-service>/render`
- Header: `Authorization: Bearer <same secret>`
- Body (JSON, built by a small Code node beforehand from
  `Image Binary to Data URL`'s output + the job's `designBible` /
  `assetHeadline` / `assetOverlayTextJson` fields):
  - `imageBase64`: strip the `data:...;base64,` prefix off
    `x.imageDataUrl`
  - `design`: the matched row from `16_DESIGN_BIBLE` (read live, see
    the earlier recommendation to add
    `ranges=16_DESIGN_BIBLE!A1:W100` to `Load Jobs & Page Folders`)
    with `Design Template JSON` parsed from its JSON string into an
    object
  - `headline`: `x.assetHeadline`
  - `overlayText`: `JSON.parse(x.assetOverlayTextJson)`
- Response format: File (binary), property name `data` — same as the
  old `Download Cloudinary Render` node produced, so everything
  downstream (`Attach Rendered PNG`, `Upload Rendered Image`, hash
  checks in `Collect Rendered Files`) keeps working unchanged.
